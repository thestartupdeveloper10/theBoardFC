Section 01

Project Overview

theBoard is a dual-audience web application serving both football fans (public) and club administrators (private). It manages players, fixtures, news, statistics, and fan engagement from a single codebase.

⚽

Fan Portal

Public-facing pages for fixtures, team roster, player stats, news, and a fan zone.

Public
🛡️

Admin Dashboard

Protected CMS for managing players, matches, news, contacts, and team statistics.

Admin Only
🗄️

Supabase Backend

PostgreSQL database, row-level security, file storage, and built-in auth.

Backend-as-a-Service
📧

EmailJS

Transactional email delivery for contact-form notifications without a backend server.

External Service
💡
JAMstack Architecture

theBoard follows a BFF-less (Backend-for-Frontend-less) JAMstack pattern. The React SPA communicates directly with Supabase via its JavaScript client — there is no custom API server layer between the browser and the database.

No Custom API Server

The SPA communicates directly with Supabase — there is no Express/Fastify/NestJS middleware layer. Security is enforced via Supabase Row Level Security (RLS) policies in the database, not application-level guards.

🔴 Auth Context — What it holds

  • session — Supabase session object (JWT)
  • user — Authenticated user metadata
  • isAdmin — Role check from profiles.role
  • isLoading — Initial auth resolution state

Auto-logout fires after 1 hour of inactivity tracked via DOM events (mousedown, scroll, keypress, touchstart).

🔵 QueryClient — Cache strategy

  • staleTime: 5 min — No refetch for 5 minutes
  • retry: 1 — One retry on failure
  • refetchOnWindowFocus: false — No aggressive refetch
  • Cache invalidation — Triggered after every mutation via invalidateQueries

Security Note

All VITE_* variables are exposed to the browser bundle at build time. The Supabase anon key is safe to expose — it is public by design. Real security is enforced via Row Level Security (RLS) policies in PostgreSQL, not by keeping the key secret.
